## Goal

Refactor `@frehilm/ordna-core` so the storage layer becomes pluggable. Today every UI calls file-based functions in `store.ts` directly. We want to introduce a `TaskProvider` interface; the existing file logic becomes the default `FileTaskProvider`. After this refactor, third-party plugins like `@frehilm/ordna-jira` and `@frehilm/ordna-linear` can implement the same interface so the TUI/Web work unchanged against any backend.

The end-state for users:

```yaml
# .ordna/config.yaml
provider: jira              # default = files
jira:
  baseUrl: https://acme.atlassian.net
  email: fredrik@helo.se
  apiTokenEnv: JIRA_TOKEN
  projectKey: ENG
  jql: "sprint in openSprints()"
```

Same TUI, same Web kanban, different source of tasks.

## Acceptance Criteria

- [ ] `TaskProvider` interface exported from `@frehilm/ordna-core`
- [ ] Existing file logic moved into `FileTaskProvider` with no behavioral change
- [ ] All existing tests still pass without modification
- [ ] `createContext()` resolves the configured provider via `.ordna/config.yaml` (`provider: file` is the default)
- [ ] `Task` type made backend-agnostic: `filePath` and `rawContent` become optional
- [ ] `commitTasks` becomes optional on the interface (no-op for non-file providers)
- [ ] `watchTasks` interface allows file watcher OR webhook/polling implementations
- [ ] Documentation: provider interface API, plus a "writing a provider" guide
- [ ] Published as `@frehilm/ordna-core@0.2.0` (minor bump — additive, no breaking changes for current consumers)

## Notes

### Architecture

```
@frehilm/ordna-core           defines TaskProvider interface
  ↳ FileTaskProvider          built-in default — markdown files in tasks/
@frehilm/ordna-jira           ← implements TaskProvider (separate package)
@frehilm/ordna-linear         ← implements TaskProvider (separate package)

@frehilm/ordna-cli + ordna-web    consume TaskProvider, agnostic to backend
```

Proposed interface shape:

```ts
interface TaskProvider {
  list(opts?): Promise<Task[]>;
  get(id): Promise<Task | null>;
  create(input): Promise<Task>;
  update(id, patch): Promise<Task>;
  move(id, status): Promise<Task>;
  delete(id): Promise<void>;
  watch(cb): () => Promise<void>;   // file: chokidar; remote: webhook+poll
  commit?(msg?): Promise<void>;     // optional, file-only
}
```

Today's `listTasks(ctx)` etc. become thin wrappers that delegate to `ctx.provider.list()`. Existing call sites stay backwards-compatible.

### Schema mismatch (lossy)

Remote trackers carry concepts Ordna doesn't:

| Concept                | Ordna    | Jira / Linear              |
|------------------------|----------|----------------------------|
| Multi-assignee         | no       | yes                        |
| Custom statuses        | yes      | yes (workflows)            |
| Sprints / cycles       | no       | yes                        |
| Epics / parent links   | no       | yes                        |
| Comments               | no       | yes                        |
| Custom fields          | `extra_frontmatter` | many, structured |
| Estimation / story pts | no       | yes                        |
| Reporter ≠ assignee    | no       | yes                        |

Plan: writes are restricted to fields Ordna already understands (status, assignee, priority, tags, depends_on, sections). Remote-only fields are surfaced as read-only chips in the UI ("Jira: 3 SP, in Sprint 47") so users see them without Ordna having to model them.

### Operational concerns for remote providers

- **Auth**: API token via env var, same convention as `ORDNA_AGENT_HOOK_HEADERS`.
- **Rate limits**: aggressive read caching. Linear's GraphQL is generous; Jira's REST is rate-limited.
- **Offline**: fail loudly with a clear error. Optional later: on-disk cache so the board renders stale rather than nothing.
- **Live updates**: webhooks if the host can expose a public URL; otherwise default to polling at a configurable interval (e.g. 30s).
- **Depends_on gate**: remains a client-side rule. Remote providers may also reject the transition; both errors surface to the user.

### Positioning trade-off

Today's README says "tasks are markdown files in git". With remote providers, that's only true for `provider: file`. The framing shifts from "files are the API" to "Ordna is the UI; files are one backend, the simplest one." Decide on README phrasing as part of this work.
