## Goal

Make it possible to manage a Kanban board in JIRA instead of the git-file-based store. Same TUI / Web UI, swappable backend.

This is **Phase 2** of the plan in T-018: ship `@frehilm/ordna-jira` as a separate package implementing the `TaskProvider` interface that T-018 introduces.

## Acceptance Criteria

- [ ] New package `@frehilm/ordna-jira` published
- [ ] Implements `TaskProvider` interface from `@frehilm/ordna-core` (post-T-018)
- [ ] Auth via env var (e.g. `JIRA_TOKEN`) with optional headers config — reuses the convention from `ORDNA_AGENT_HOOK_HEADERS`
- [ ] Config block in `.ordna/config.yaml`:
  ```yaml
  provider: jira
  jira:
    baseUrl: https://acme.atlassian.net
    email: ...
    apiTokenEnv: JIRA_TOKEN
    projectKey: ENG
    jql: "sprint in openSprints()"
  ```
- [ ] Read-only first: `list`, `get`, `watch` (polling-based, configurable interval) work end-to-end in the TUI and Web
- [ ] Write support phase: `move` (status transitions), `update` (title / assignee / priority / tags), `create`, `delete` — restricted to fields Ordna already understands
- [ ] Schema mismatch handled per T-018: read remote-only fields (sprint, story points, epic link) into a read-only chip strip in the modal; never write them
- [ ] Rate-limit-aware: aggressive read caching, exponential backoff on 429
- [ ] Offline behavior: clear error messaging; on-disk cache so the board renders stale rather than nothing (stretch)
- [ ] Documentation: `packages/jira/README.md` covers setup, supported fields, lossy fields, rate-limit notes
- [ ] depends_on gate continues to work as a client-side rule on top of JIRA's transition rules

## Notes

See the architecture and schema-mismatch table in T-018 §"Notes". Don't model JIRA-only concepts (sprints, epics, comments, multi-assignee) in `Task` itself — surface them via a read-only `extras` blob the UI displays but doesn't edit.

Live updates: webhooks if the host can expose a public URL; default to polling at 30s otherwise (configurable).
