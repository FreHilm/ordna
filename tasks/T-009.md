## Goal

Make it possible to manage a Kanban board in Linear instead of the git-file-based store.

This is **Phase 4** of the plan in T-018: ship `@frehilm/ordna-linear` reusing the patterns proven by `@frehilm/ordna-jira` (T-008).

## Acceptance Criteria

- [ ] New package `@frehilm/ordna-linear` published
- [ ] Implements `TaskProvider` interface from `@frehilm/ordna-core` (post-T-018)
- [ ] Auth via env var (e.g. `LINEAR_API_KEY`)
- [ ] Config block in `.ordna/config.yaml`:
  ```yaml
  provider: linear
  linear:
    apiKeyEnv: LINEAR_API_KEY
    teamId: ENG
    filter: "state.type IN (started, unstarted)"
  ```
- [ ] Read-only first: `list`, `get`, `watch` work end-to-end via Linear's GraphQL API
- [ ] Write support phase: `move`, `update`, `create`, `delete` for fields Ordna understands
- [ ] Schema mismatch handled per T-018: cycle / project / parent-issue surfaced as read-only chips
- [ ] Documentation: `packages/linear/README.md`
- [ ] Reuses the field-mapping helpers / `extras` blob convention introduced by T-008

## Notes

Linear's GraphQL is significantly nicer than JIRA's REST — query batching, no rate-limit headaches in normal use, real-time subscriptions available. Build T-008 first (it's the harder of the two), then T-009 should mostly be a different transport over the same `TaskProvider` shape.

Real-time: Linear supports GraphQL subscriptions, which we should prefer over polling when available.
